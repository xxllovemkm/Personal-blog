## 为什么要把 data engineering 类比成 software engineering

做领域模型时，一个很常见的流程是：收一批 raw corpus，让 LLM 合成 SFT 数据，训练一次，再看 benchmark 分数。问题在于，一旦模型错了，我们很难回答最关键的问题：到底是哪个概念没学到，哪条推理链断了，应该补什么数据？

这就是 ProDa 想解决的问题。我们不是把数据看成越多越好的燃料，而是把它看成源代码。训练就是编译，benchmark 是单元测试，模型失败就是运行时暴露出来的 bug。既然是 bug，就应该能定位、能修复、能回归测试。

## 最核心的变化：benchmark 不只是打分器

ProDa 里 benchmark 的角色和传统评测不一样。传统 benchmark 更像排行榜，告诉你模型现在几分；ProDa 里的 benchmark 更像 unit tests，必须能反向指导数据修复。

为了做到这一点，训练数据和评测数据不能各做各的。我们先从 raw corpus 中抽取结构化知识，再基于同一套知识结构生成 benchmark、SFT data 和 repair data。这样模型在某道题上失败时，系统可以追踪到相关的 L1 concept、L2 factual statement 或 L3 reasoning chain。

<Image src={intro} alt="ProDa 从开放式数据工程转向 specification、implementation、debugging 闭环" />

这个设计看起来绕了一圈，但它解决了一个非常实际的问题：如果评测样本和训练样本没有共同的知识锚点，失败就只能被描述成“这类题不行”；有了共享知识结构，失败才可能被描述成“这个概念边界不清”或“这个因果链条缺了中间步骤”。

## 数据不是随便合成，而是按 CORE 原则组织

ProDa 里我们把数据合成要求压成 CORE：

- Contextualized：样本要有足够上下文，不能脱离原始文档瞎编。
- Organized：知识要分层组织，能复用、能追踪。
- Rigorous：推理链要严格，不能只生成表面问答。
- Evolving：训练后根据失败继续修复，而不是一次性生成完。

最终 ProDalib 覆盖 16 个学科，包含 227k key concepts、16k evaluation items 和 160k synthesized training samples。这里的重点不是数字本身，而是每个 artifact 都能回到同一个知识来源。

## 做这个项目时最容易被低估的地方

最难的不是生成问答，而是生成“可诊断”的问答。

如果一道题只是问一个孤立事实，模型答错了也很难判断是概念没见过、概念混淆，还是推理能力不足。所以我们设计了不同粒度的知识对象：L1 抽概念，L2 抽原子事实关系，L3 抽多步推理链。这样诊断时才能区分 concept gap 和 capability deficit。

另一个痛点是 repair data 不能只围着错题打转。只补失败点会导致局部过拟合，甚至忘掉原来会的内容。因此后续数据混合里要引入 error-proportional allocation 和 replay，并且用 L2 ID disjoint 约束避免 replay 样本和 repair 样本重复同一知识节点。

## 实验真正想证明什么

ProDa 的实验不是想证明“我们的合成数据比某个 baseline 多一点分”，而是想证明闭环修复本身有价值。

<Image src={scaling} alt="ProDa 在不同数据规模下和常规合成数据方法的性能对比" />

从 scaling 图里最重要的信号是 sample efficiency：ProDa V2 通过诊断修复，在较少样本下就能接近或超过更大规模的常规数据方案。这符合我们的直觉：如果知道 bug 在哪里，补丁数据应该比盲目扩充更有效。

case study 也服务于同一个论点。模型失败后，系统不是泛泛地再生成一批“相关题”，而是根据诊断报告生成针对 concept gap 或 reasoning deficit 的 patch。

## 我对这个方向的判断

ProDa 对我来说最重要的启发是：未来的模型能力建设不应该只讨论 model scaling，也要讨论 data lifecycle。

如果训练数据不可追踪、评测不可诊断、失败不可修复，那么所谓 self-improving 很容易退化成“模型自己生成更多东西再训练”。ProDa 想把自我改进落在更硬的工程闭环上：先定义 specification，再编译数据，再运行测试，再根据失败写 patch。

它和 PaperFlow 也有内在联系。PaperFlow 管理的是研究者每天面对的信息流，ProDa 管理的是模型从领域语料中吸收知识的工程流。一个面向人的科研助手，一个面向模型的数据工程，但都在解决同一个问题：知识不是一次性摄入的，而是需要长期组织、反馈和更新。
