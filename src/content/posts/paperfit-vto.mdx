## 这个项目最初不是想做「LaTeX 生成」

PaperFit 的出发点其实很日常：一篇论文已经能编译了，但 PDF 仍然不像一篇可以提交的论文。

浮动体离正文引用太远、表格伸出栏宽、公式轻微溢出、最后一页空得像缺内容、模板迁移后图表宽度全乱。这些问题在 LaTeX log 里要么没有信号，要么只是一个很弱的 warning；但作者一眼就能看出来不对。

所以我们没有把目标定义成“从自然语言生成 LaTeX”，而是定义成更靠近投稿前真实工作流的任务：给定一个可编译项目，让系统像作者一样经历 **compile -> inspect -> edit -> recompile**，直到 PDF 在视觉上可接受，并且满足页数和模板约束。

## VTO 这个定义是被问题逼出来的

写论文时大家真正关心的不是“源码语法是否正确”，而是最终 PDF 是否达标。我们后来把这个任务叫 Visual Typesetting Optimization，简称 VTO。

这个定义有两个关键约束。

第一，优化对象是渲染后的页面，不是源码本身。很多缺陷只有 page image 才能看见，比如表格虽然源码合法，但视觉上已经越过 column；或者图虽然没有报错，但把阅读顺序打断了。

第二，修复必须是受约束的。排版修复很容易出现伪解：随手 `\vspace{-10pt}`、粗暴 `\resizebox`、强行 `\newpage`，短期看起来有效，但会破坏模板规范、字体比例或后续页面。PaperFit 里我们把这些操作显式列为低优先级或禁用操作，把“内容保真”和“模板一致”放在视觉优化之前。

## Benchmark 最难的是让缺陷真实

PaperFit-Bench 最后包含 200 篇论文、10 种 venue template、13 类扰动，覆盖空间利用、浮动体、表格宽度、溢出、跨模板迁移五大类。

<Image src={perturbation} alt="PaperFit-Bench 中五类扰动和 13 种具体排版缺陷的分布" />

这里比较容易做错的一点是把 benchmark 做成玩具例子：每个 case 只塞一个很干净的 overfull table，然后让模型改回来。真实投稿不是这样。真实问题往往是多缺陷叠加：模板迁移导致页数漂移，浮动体被挤走，表格同时越界，最后一页又出现大块留白。

所以我们更强调 mixed disturbance。每个 case 都保留扰动 manifest、编译日志、渲染页面和目标页数，方便之后判断模型到底修了什么，也防止模型只靠“让它能编译”糊弄过去。

## PaperFit 的关键不是看图，而是有组织地闭环

一开始我们也验证过一个很自然的 baseline：把源码、log、页面截图都给视觉模型，让它多轮修改。结果它确实比纯文本强，但不稳定。模型会修掉一个问题，又引入另一个问题；或者只要 PDF 能 render，就提前停止。

PaperFit 真正加入的是三件事：

1. 多源证据整合：source、log、PDF、page image 各自负责不同信号。
2. 受约束修复策略：哪些操作优先，哪些操作禁止，哪些对象不能动。
3. checklist-gated validation：每次修改后重新编译、渲染、检查全篇，而不是只看局部。

<Image src={pipeline} alt="PaperFit 的多源证据、受约束修复和 checklist-gated validation pipeline" />

这个设计听起来像工程细节，但正是 reviewer 会追问的地方：你是不是只是把截图丢给 VLM？实验表明不是。VisualMR 也能看页面、也能多轮修，但缺少结构化 defect records、repair constraints 和 gatekeeper 后，页数约束和全局稳定性都会明显变差。

## 实验里最能说明问题的两个发现

第一个发现是：视觉反馈必要，但不充分。PaperFit 在编译/渲染成功、视觉质量和 page-budget compliance 上都显著优于 baselines；最有信息量的对比不是和完全看不见图的模型比，而是和同样能看图的 naive visual agent 比。

<Image src={modelComparison} alt="PaperFit 在多维 VLM 评分上的模型对比" />

第二个发现是：自动视觉评价在这个任务上是可用的。我们用 human/VLM correlation 做了检查，整体 Spearman 相关达到 0.8571。这很重要，因为如果评价本身不可信，VTO 就很难扩展成 benchmark。

<Image src={correlation} alt="PaperFit 中 VLM 评分和人工评分的相关性" />

当然，PaperFit 也不是万能的。hard case 里如果同时有 5-8 个扰动，page-budget hit rate 会下降到约 70%。这说明最后一公里排版优化仍然很难，尤其当浮动体密度、模板差异和页数预算同时施压时。

## 回过头看这个工作的价值

我觉得 PaperFit 最重要的不是“做了一个排版工具”，而是把文档智能里一个长期被忽略的阶段单独定义出来了。

过去很多 document automation 工作停在 structural formatting：把 Word、Markdown、PDF 或自然语言变成可编译 LaTeX。但科研作者真正痛苦的地方往往发生在之后：论文已经能编译了，距离投稿却还有大量视觉修复。

PaperFit 试图补上这段缺口。它和 ProDa、PAGER 有一个共同点：都不是让模型自由生成，而是把模型放进一个有约束、有验证、有反馈的工程系统里。ProDa 把数据当成可测试源代码，PAGER 把 GUI 操作当成可验证几何执行，PaperFit 则把 PDF 页面当成最终验收对象。

如果后续继续做，我更想把它接到真实作者工作流里：Overleaf、arXiv 编译、会议模板检查、论文主页生成。因为这个任务只有进入真实 deadline 场景，才会暴露出最有价值的问题。
