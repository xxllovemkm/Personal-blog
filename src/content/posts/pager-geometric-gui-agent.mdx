## 从 GGBench 到 PAGER：问题往前推进了一步

GGBench 之后，我们很自然地遇到下一个问题：模型能不能不只是写 GeoGebra 代码，而是在真实 GUI 里一步步把几何图形画出来？

这件事比普通 GUI agent 难很多。网页自动化里点按钮是 region-tolerant 的，只要点在按钮范围内就算成功；几何构造是 point-precise 的，点 A 偏几像素，后面的线、圆、交点、角度都会跟着偏。模型可能完全理解“下一步应该画对角线”，但执行时点错端点，最终图形就失效。

这就是我们在论文里说的 Semantic-Execution Gap：语义上知道做什么，执行上做不出来。

## PAGE Bench：不能只看最后图像像不像

为了把这个问题量化，我们构建了 PAGE Bench。数据规模是 4,906 个几何问题、53,277 个高层 construction tasks、224,497 条低层 GUI actions。

<Image src={dataPipeline} alt="PAGE Bench 从几何题筛选、任务序列生成、GUI 执行到质量过滤的数据 pipeline" />

这个数据集不是静态的“题目 + 答案图”。每个轨迹里都保存了问题、当前截图、历史动作、下一步 action、执行反馈、坐标和空间标注。原因很简单：如果只看最终图，很多失败无法归因。我们需要知道模型到底是选错工具、点错位置、参数漂移，还是前面一步错误导致后面全部崩掉。

数据构建中最麻烦的是把抽象几何步骤变成可复现的浏览器动作。GeoGebra 里的每个 construction step 都要拆成工具选择、canvas 操作、输入参数、边界检查和重试。这个过程迫使我们把“几何推理”真正落到 pixel-level GUI execution 上。

## PAGER 的设计：规划管依赖，执行管像素

PAGER 的结构其实很朴素：不要让一个模型在同一层里同时解决所有问题，而是把构造拆成 dependency-structured planning 和 pixel-level execution。

<Image src={modelArch} alt="PAGER 的 topology-aware planning 和 pixel-level execution 框架" />

规划模块负责把几何题拆成有拓扑依赖的任务序列。几何对象不是独立生成的：线依赖点，交点依赖两条线，圆依赖圆心和半径，后续对象又依赖前面所有状态。这个 dependency graph 是防止“画着画着世界变了”的关键。

执行模块负责把每个子任务转成 GUI action。我们先做 pixel-grounded SFT，让模型学到可执行动作语法；再做 precision-aligned RL，用 action-type correctness、parameter accuracy 和 rendered geometric validity 对齐训练目标。

这里强化学习真正优化的是点级执行，而不是泛泛奖励“任务成功”。如果 reward 只看动作类型，模型会学会选择正确工具，但坐标仍然漂；如果 reward 只看局部坐标，又可能破坏操作顺序。实验里两个 reward 是互补的。

## 最刺眼的结果：动作类型很高，任务成功很低

这篇论文里最能说明问题的数字是：Claude-Sonnet-4.6 的 Action Accuracy 到 95.85，GPT-5.4 和 Gemini-3.1-Pro 也分别到 88.04 和 89.18，但 Task Success 只有 1.11、0.56 和 5.82。

换句话说，它们大多知道应该选什么动作，却不能稳定把动作落到正确几何参数上。

PAGER 的 Task Success 达到 23.78，大约是最强 general baseline Gemini-3.1-Pro 的 4.1 倍；Step Success 也从 GUI-specialized agents 的低于 9% 提到 62.20。这个提升不是因为模型突然更懂几何，而是因为训练目标终于对准了“点级执行 + 依赖传播”这个瓶颈。

## 自动评测和人类判断要对齐

几何 GUI 任务还有一个坑：如果指标只看截图像素相似，可能会奖励错误图形；如果只看动作字符串，又会忽略等价构造路径。所以 PAGE Bench 的指标同时看过程、参数和最终几何有效性。

我们用人工评分检查了自动评测，相关性达到 0.9397。这个结果对 benchmark 很关键，因为它说明指标不是在优化一个偏离人类判断的 proxy。

## 这篇工作在研究线索里的位置

PAGER 和 GGBench 是一条线上的两个阶段。

GGBench 问的是：模型能不能把语言几何题转成可验证的构造？PAGER 进一步问：如果不给你代码执行器，而是给你真实 GUI，你能不能把构造过程精确操作出来？

这也连接到更大的 GUI agent 问题。现在很多 agent benchmark 默认 GUI 是离散组件集合，但实际生产工具不总是这样：CAD、标注平台、图形编辑器、科学可视化都要求连续空间控制。PAGER 想证明的是，GUI agent 如果要进入这些专业工具，必须从 region grounding 走向 precision-sensitive control。

后续如果继续推进，我会更关心两个方向：一是把拓扑规划和显式几何求解器结合，减少模型在坐标上的自由漂移；二是把这个执行框架迁移到更真实的专业软件，而不只是 GeoGebra。
